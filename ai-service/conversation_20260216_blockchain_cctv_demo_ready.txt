================================================
SYSTEM ARCHITECTURE:
================================================

Data Flow:
1. User initiates webcam capture via frontend
2. Frontend calls AI service /api/ai/start-stream
3. AI service captures video frames, saves recording
4. Gemini AI analyzes frames, generates forensic report
5. AI service calculates SHA-256 hash of video
6. Backend receives hash via /api/record endpoint
7. Backend stores hash in SQLite (status='pending')
8. Backend calls smart contract logEvidence() function
9. Blockchain confirms transaction, emits event
10. Backend updates SQLite (status='confirmed')
11. Frontend displays evidence log on dashboard

Verification Flow:
1. User uploads video file via frontend verifier
2. Frontend calculates SHA-256 hash client-side
3. Frontend calls backend /api/verify endpoint
4. Backend checks SQLite cache (Level 1)
5. Backend queries blockchain smart contract (Level 2)
6. Returns verification result (Verified/Tampered/Pending)

KNOWN LIMITATIONS:
================================================

1. Local Development Only
   - Hardhat node not suitable for production
   - No persistent blockchain state between restarts

2. Security Considerations
   - Private keys exposed in .env (acceptable for demo)
   - No authentication on API endpoints
   - CORS wide open for localhost

3. Performance
   - Synchronous blockchain calls may block
   - No queue system for high-volume uploads
   - SQLite suitable for demo, not production scale

4. Error Handling
   - Limited retry logic for blockchain failures
   - No dead letter queue for failed uploads
   - Camera stream may timeout without warning
